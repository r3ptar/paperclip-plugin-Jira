## Architecture

This is a **Paperclip Plugin SDK** plugin. The SDK provides `definePlugin()`, `runWorker()`, and typed registries for tools, webhooks, jobs, UI slots, and config schemas.

### Key Layers

- **`src/jira/`** -- HTTP client layer: `JiraClient` (circuit breaker, 429 backoff, 401 refresh), `TokenManager` (OAuth 2.0 3LO, API token, PAT), Jira REST API types, ID/key validation
- **`src/services/`** -- Business logic services (issues, projects, boards, sprints, users, search, identity) that wrap `JiraClient`
- **`src/sync/`** -- Bidirectional sync engine: status mapping, conflict resolution (3 strategies), JQL-based reconciliation
- **`src/tools/`** -- 15 agent tool handlers (each tool is a separate file, registered in the manifest)
- **`src/webhooks/`** -- Jira webhook handler for issue lifecycle events (created, updated, deleted)
- **`src/ui/`** -- React components: SettingsPage, SetupWizard, DashboardWidget, IssueTab, ProjectTab, plus sub-components (StatusMappingEditor, AgentIdentityEditor, ConnectionStatus, JqlPreview, WizardStep)
- **`src/manifest.ts`** -- Plugin manifest (tools, jobs, webhooks, UI slots, config schema)
- **`src/worker.ts`** -- Plugin entrypoint: `definePlugin()` / `runWorker()`, service initialization, handler registration
- **`src/constants.ts`** -- All registries (TOOL_NAMES, JOB_KEYS, WEBHOOK_KEYS, SLOT_IDS), `JiraConfig` type, `DEFAULT_CONFIG`

### Critical Design: Transition-Based Workflow Sync

Jira does **not** allow setting status directly -- you must execute a valid workflow transition. This is the biggest difference from the M365 plugin:

1. Fetch available transitions via `GET /issue/{key}/transitions`
2. Find transition whose `to.name` matches the target status
3. Fall back to matching by status category (`new`/`indeterminate`/`done`)
4. Execute via `POST /issue/{key}/transitions`

Status mappings are stored in config (not hardcoded) because Jira workflows vary per project. The `StatusMappingEditor` UI component fetches the project's workflow and lets users configure mappings.

### Cloud vs Server/Data Center

- Cloud: `/rest/api/3` (ADF rich text), OAuth 2.0 3LO
- Server/DC: `/rest/api/2` (wiki markup), PAT auth
- Agile API: `/rest/agile/1.0` (same for both)
- `JiraClient` selects base path from `config.deploymentMode`
- `JiraIssueService` handles description format differences in create/update

### Sync Scope

`config.syncJql` (a JQL query) defines which issues are tracked. Reconciliation uses this JQL directly for batch fetching rather than syncing all issues.
